# LacquerTutor 产品方案 / PRD

配套工程交付规格见 [ENGINEERING_DELIVERY_SPEC_CN.md](D:/qiyupaper/lacquertutor/ENGINEERING_DELIVERY_SPEC_CN.md)。

## 1. 产品定义

### 1.0 产品立场
这个产品**借用论文的洞察，不照搬论文的机制**。

保留的，是论文真正有价值的部分：

- 漆艺是强过程、强上下文、强不可逆的任务。
- 用户经常问得不完整。
- 系统必须在关键节点主动补全信息。
- 输出应该是可执行方案，而不是泛化建议。

不必绑定的，是论文里的具体实现形式：

- 不一定非要用 VoI 公式。
- 不一定非要做研究范式里的多模块命名。
- 不一定非要把评测结构原样搬进产品。

产品可以更务实：

- 用 **prompt + 规则 + 工作流** 做动态追问。
- 用 **安全门控** 替代学术术语暴露。
- 用 **项目工作台** 替代论文中的实验界面。
- 用 **真实可执行交付物** 替代研究 JSON 输出。

### 1.1 产品一句话
**LacquerTutor** 不是一个泛化问答机器人，而是一个面向漆艺制作与排障的**过程安全智能体产品**：它会主动追问关键工艺变量，基于证据给出可执行方案，并在不可逆步骤前强制“安全停步”。

### 1.2 产品要解决的问题
漆艺学习里最大的痛点不是“没有教程”，而是：

- 用户拿到的是泛化配方，不知道是否适用于自己的材料、环境和阶段。
- 新手描述问题往往不完整，只说“发白”“不干”“不亮”，却缺少湿度、漆种、层厚、前序工艺这些关键变量。
- 很多错误一旦跨过步骤边界就不可逆，代价是返工、报废，甚至健康风险。
- 现有内容平台擅长“展示结果”，不擅长“保障过程”。

### 1.3 核心产品主张
LacquerTutor 的核心不是“回答得像专家”，而是：

1. **先问对问题，再给方案。**
2. **先确认能不能做，再告诉你怎么做。**
3. **每个关键建议都能追溯到证据。**
4. **在不可逆步骤前，宁可停，也不瞎猜。**

## 2. 目标用户

### 2.1 第一目标用户
**漆艺初学者与进阶学习者**

- 来自高校工艺美术、非遗、手作课程。
- 能看懂基本术语，但对材料体系、环境控制、排障判断不稳定。
- 有明确任务，例如“做木胎底漆”“出现发白该怎么办”“如何达到镜面光泽”。

### 2.2 第二目标用户
**工作坊 / 教学场景中的老师与助教**

- 需要批量回答重复问题。
- 需要规范流程与安全边界。
- 需要把隐性经验转成可复用的教学流程。

### 2.3 第三目标用户
**中小型漆艺工作室**

- 需要标准化项目 SOP。
- 需要留存“这件作品当时怎么做”的过程记录。
- 需要减少返工和试错损耗。

## 3. 产品形态

### 3.1 我们做的不是

- 不是通用聊天助手。
- 不是只会推荐教程的视频平台。
- 不是只会给配方的检索工具。
- 不是“你问什么我答什么”的被动问答框。

### 3.2 我们做的是什么
一个围绕“项目执行”组织的漆艺智能工作台，核心由四个层级组成：

1. **诊断对话层**
   负责识别任务类型、抽取上下文、主动补问缺失变量。
2. **证据检索层**
   从工艺知识库中找出与当前阶段、故障模式、材料条件匹配的证据卡。
3. **计划合同层**
   将建议输出为结构化“可执行计划合同”，而不是一段散文。
4. **执行监护层**
   在关键步骤前设置门控、检查点、风险警告和应急分支。

### 3.3 最适合的产品终端
优先级从高到低：

1. **Web 应用**
   适合结构化多栏界面，能同时展示对话、变量状态、证据卡和计划合同。
2. **微信/小程序轻量入口**
   适合咨询和项目跟进，但不适合复杂合同编辑。
3. **面向教学机构的后台版**
   适合知识库管理、案例沉淀、班级项目跟踪。

## 4. 核心使用场景

### 场景 A：新项目开工
用户上传参考图或描述目标效果，例如“我想在木胎上做亮光黑漆底”。

系统流程：

1. 判断这是 planning 任务。
2. 抽取已知变量，如基底、目标效果、已有工具。
3. 识别缺失的安全门控变量，如漆种、湿度、固化方式、防护。
4. 按风险优先级提问，而不是一次问完所有信息。
5. 检索相应证据。
6. 生成“开工合同”，包含步骤、参数、检查点、不可逆警告和失败分支。

### 场景 B：问题排查
用户说“我的漆面固化两天后发白发雾，要不要打磨重涂？”

系统流程：

1. 判断这是 troubleshooting 任务。
2. 将“发白发雾”规范化为 `haze / whitening`。
3. 识别下一个不可逆动作是“打磨重涂”。
4. 在湿度、层厚、固化状态未确认前阻止跨步。
5. 先生成“安全停步 + 检查顺序”，而不是立刻给补救动作。
6. 条件确认后再输出修复合同。

### 场景 C：跟做与复盘
用户按合同执行后，记录每一步结果，形成个人工艺档案。

系统流程：

1. 保存本次项目的已填变量、证据、合同和实际偏差。
2. 用户可标记“成功”“返工”“失败原因”。
3. 系统将成功案例转为可复用模板。
4. 下次相似项目优先复用“你自己的工艺经验”。

## 5. 产品核心机制

### 5.1 主动澄清
不是等用户把信息说全，而是系统根据风险判断现在最该问什么。

产品层面的体现：

- 左侧展示“关键变量进度条”。
- 对每个未确认变量展示优先级和原因。
- 每次只问 1 个最关键问题，减少打扰。

这里的底层实现可以是三种方式，产品不需要绑定单一算法：

1. **Prompt 驱动**
   让模型判断“为了给出下一步安全建议，还缺什么信息”。
2. **规则驱动**
   对漆种、湿度、基底、固化方式等硬门槛做显式约束。
3. **排序器增强**
   需要时再加入 VoI 或其他优先级算法，提高追问质量。

也就是说，VoI 是一个可选增强，不是产品成立的前提。

### 5.2 不可逆门控
一旦下一步可能带来不可逆后果，系统必须先确认门控变量，否则停止推进。

产品层面的体现：

- 用显式的 `Blocked` / `Safe to Proceed` 状态。
- 对用户说清楚“为什么不能继续”。
- 给出当前可做的安全动作，而不是只说“不行”。

### 5.3 证据卡驱动
关键建议必须能追溯到来源，而不是模型即兴生成。

产品层面的体现：

- 每个合同步骤挂载证据卡。
- 证据卡显示适用阶段、故障模式、摘要、来源。
- 用户可以展开“为什么系统这么建议”。

### 5.4 合同式输出
输出不是自然语言长段落，而是可执行合同。

合同固定包含 6 个部分：

1. 已知条件与假设
2. 缺失关键变量
3. 操作步骤
4. 高风险警告
5. 检查点
6. 应急预案

### 5.5 项目记忆
产品不能只做单轮问答，必须形成持续项目状态。

产品层面的体现：

- 一个作品就是一个项目。
- 所有提问、回答、证据和合同都留档。
- 用户可在第二天回来继续，不必重说一遍。

## 6. 产品信息架构

### 6.1 一级导航

1. **首页**
   新建项目、最近项目、常见问题入口。
2. **项目工作台**
   产品核心页。
3. **知识库**
   教程、证据卡、工艺术语、案例。
4. **模板库**
   常见工艺流程模板和个人复用模板。

### 6.2 项目工作台结构
项目工作台推荐采用三栏布局：

1. **左栏：上下文与门控**
   阶段、故障模式、已填变量、未填变量、VoI 优先级、门控状态。
2. **中栏：对话与追问**
   用户输入、系统追问、即时解释、上传图片/工序记录。
3. **右栏：证据与合同**
   证据卡列表、步骤合同、检查点、风险警告、导出按钮。

## 7. 功能设计

### 7.1 MVP 必做功能

#### 模块 A：项目咨询入口
- 支持“做什么”和“出了什么问题”两种入口。
- 支持文本输入。
- 支持上传参考图或问题图。

#### 模块 B：变量采集与追问
- 自动抽取已知变量。
- 展示缺失关键变量。
- 按优先级单轮追问。
- 用户可手动填写环境、材料、时间等字段。
- 追问策略以 **prompt playbook + 安全规则** 为主，不依赖复杂学术评分器也能工作。

#### 模块 C：证据检索
- 基于阶段、故障模式、材料条件进行检索。
- 给出 3 到 5 条最相关证据卡。
- 每条证据卡有摘要和适用边界。

#### 模块 D：计划合同生成
- 输出 6 段式合同。
- 支持一键导出 Markdown / PDF。
- 支持“安全停步合同”。

#### 模块 E：执行跟踪
- 用户可勾选步骤完成状态。
- 关键检查点要先确认才能进入下一步。
- 支持记录照片、备注、环境读数。

### 7.2 第二阶段增强功能

- 作品项目时间线
- 环境传感器接入，自动获取温湿度
- 图像辅助诊断
- 用户个人工艺档案
- 老师批注 / 工作室复核
- 模板复用与案例推荐

### 7.3 第三阶段产品化能力

- 机构私有知识库
- 多角色权限
- 课程任务布置与批改
- 耗材与工具推荐
- 作品质量追踪与返工分析

### 7.4 这款产品真正可用时的 AI 能力形态
面向上线产品，推荐的不是“一个复杂研究代理”，而是 4 个可控的 prompt workflow：

1. **任务理解 Prompt**
   判断用户是在“想做某效果”还是“出了某问题”，并抽取当前阶段。
2. **缺口追问 Prompt**
   判断如果现在要给下一步建议，还缺哪些必要信息。
3. **计划生成 Prompt**
   把当前上下文变成步骤、检查点、风险提示和补救方案。
4. **安全审查 Prompt / Rule Engine**
   判断是否存在不能直接放行的步骤。

这个组合足够支撑一个可上线的 V1 产品，而且比一开始做重型 agent 更稳。

## 8. 与论文相比，产品化必须新增的东西

论文解决的是“方法成立”，产品还要解决“为什么愿意长期用”。

### 8.1 从单轮实验任务，升级为项目生命周期
论文关注一次任务生成。
产品必须覆盖：

- 新建项目
- 执行中跟进
- 中断后恢复
- 完成后复盘

### 8.2 从研究评测指标，升级为用户价值指标
论文里的 M1-M7 是系统能力指标。
产品需要增加：

- 返工率下降
- 用户完成率
- 每个项目平均追问轮次
- “建议被执行”的比例
- 复用模板率
- 次日/7 日留存

### 8.3 从证据引用，升级为信任体验
用户不是为了看引用编号，而是为了“敢照着做”。

所以产品层必须加入：

- 为什么问我这个问题
- 为什么现在不能继续
- 这个建议适用于哪些条件
- 如果没有仪器，我该怎么保守处理

### 8.4 从算法输出，升级为交付物
产品交付的不是一个 JSON，而是：

- 可执行清单
- 分阶段提醒
- 风险提示
- 项目记录
- 可复用模板

## 9. 商业化思路

### 9.1 商业定位
先从“专业学习工具”切入，而不是大众泛兴趣聊天产品。

### 9.2 可能的付费模式

#### 个人版
- 免费版：有限项目数、基础咨询、基础证据。
- Pro 版：无限项目、合同导出、项目复盘、图像诊断、模板库。

#### 教学机构版
- 按班级或按老师席位收费。
- 提供课程案例库、学生项目追踪、教师审核与点评。

#### 工作室版
- 私有知识库
- SOP 模板沉淀
- 多人协作
- 返工与质量统计

### 9.3 最合理的起步路径

1. 先做面向个人学习者的 MVP，验证高频需求。
2. 再进入高校工作坊，验证教学价值。
3. 最后扩展到工作室和非遗培训场景。

### 10.1 相比通用 AI 助手

- 我们会主动补问关键变量。
- 我们知道哪些步骤不可逆。
- 我们输出的是流程合同，不是散文建议。
- 我们有领域知识边界和证据约束。

### 10.2 相比教程平台

- 我们针对“我的当前状态”给建议。
- 我们能做排障，而不是只讲标准流程。
- 我们会在不安全时阻止推进。

### 10.3 相比传统工艺社区

- 社区给经验碎片。
- LacquerTutor 给结构化决策与执行流程。

## 17. 最新智能体构建思路下的产品技术路线

以下部分不是单纯“怎么实现论文 pipeline”，而是基于当前主流 agent 构建范式，对产品底层做重新定义。

### 17.1 总原则：先工作流，后自治
这类产品不适合一上来就做“完全自主、多代理自由协作”的黑盒系统。

更稳妥的方式是：

1. **先把高价值路径做成可控 workflow**
   例如：任务识别 → 关键变量补问 → 检索证据 → 生成合同 → 验证 → 人工确认。
2. **只把不确定、开放式的部分交给 agent**
   例如：追问策略、故障归因、证据选择、合同修订。
3. **在不可逆节点前强制回到显式控制流**
   不能让 agent 自己“觉得差不多了就继续”。

这比传统“一个大 prompt + 一堆工具”的方式更符合最新的 agent 实践，也更适合高风险工艺场景。

### 17.2 推荐的现代 agent 架构
建议把 LacquerTutor 产品定义为一个**受约束的 agentic workflow**，而不是一个完全自由代理。

推荐结构：

1. **Workflow Orchestrator**
   负责任务阶段切换、停步条件、人工确认、重试和审计。
2. **Intake Agent**
   识别任务类型、抽取槽位、判断当前阶段。
3. **Gatekeeper Agent**
   负责追问优先级、不可逆门控判断、决定“继续问 / 停止推进 / 进入计划”。
4. **Retrieval Agent**
   负责从知识库、案例库、规则库中找证据。
5. **Contract Planner Agent**
   生成结构化计划合同。
6. **Verifier / Policy Engine**
   用规则和评测器校验计划，不通过则打回。
7. **Human Checkpoint**
   在关键节点允许老师、助教或用户本人确认后再继续。

这里真正“自治”的只有少数几个环节，整个产品仍然由显式 workflow 托底。

如果早期要更快落地，甚至可以不拆成多 agent，而是先做成：

1. 一个主 orchestrator
2. 一套 prompt playbooks
3. 一个规则校验器
4. 一个项目状态机

这已经足够做出第一版可用产品。

### 17.3 推荐技术栈

#### 编排层
- **OpenAI Agent Builder / Agents SDK / Responses API**
  适合做可视化工作流、代码化运行时、工具调用、流式输出和 trace。
- 如果后续需要更强的长流程可恢复执行，也可以引入 **LangGraph** 作为底层状态机与 durable execution 层。

#### 连接层
- **MCP**
  用来统一接入知识库、项目数据库、传感器服务、教学后台、素材库等外部能力。

#### 工具层
- **Built-in file search**
  用于漆艺知识库、案例库、老师讲义、工艺规范等检索。
- **Web search**
  仅在查询最新法规、供应商信息、材料标准时开启，默认不要对用户咨询全量放开。
- **Remote MCP tools**
  用于温湿度传感器、教学系统、工作室库存、工单系统等。

#### 观测与优化层
- **Trace logging + trace grading**
  评估 agent 每一步是否合理。
- **Datasets / regression evals / capability evals**
  做持续回归和版本对比。

### 17.4 为什么要用 MCP，而不是继续堆自定义函数
MCP 的价值不只是“多一种接工具的方法”，而是把外部能力拆成三种标准原语：

1. **Resources**
   给模型看的上下文，例如工艺卡、案例、作品档案、材料说明。
2. **Prompts**
   可复用的领域任务模板，例如“生成安全停步合同”“做排障问询”。
3. **Tools**
   真正会触发动作的能力，例如读取传感器、写入项目记录、生成导出文件。

对 LacquerTutor 这种产品，最合理的做法不是把所有东西都做成 tools，而是：

- 知识类内容走 `resources`
- 流程模板走 `prompts`
- 外部动作走 `tools`

这样上下文、动作、模板分层更清楚，也更容易治理。

## 18. 面向产品的现代 Agent 设计原则

### 18.1 Tool-first 不是越多越好
最新实践并不鼓励“工具越多越聪明”，而是强调：

- 工具边界清晰
- 名称和参数可读
- 输入输出严格类型化
- 每个工具只做一件事
- 相似工具要命名空间隔离

对 LacquerTutor 来说，工具不应按“技术模块”拆，而应按“用户结果”拆。例如：

- `project.get_current_state`
- `project.save_checkpoint`
- `craft.read_environment`
- `craft.lookup_material_compatibility`
- `contract.generate`
- `contract.verify`

而不是泛泛的 `query_db`、`search_docs`、`execute_rule`。

### 18.2 Human-in-the-loop 要前置，不要补救式加上
在不可逆工艺场景，人类确认不是可选增强，而是核心产品能力。

必须设置人工确认的节点：

- 跨入不可逆步骤前
- 传感器数据缺失时
- 模型证据不足时
- 用户明显想跳过安全步骤时
- 诊断结果分歧较大时

### 18.3 Durable execution 是产品刚需
漆艺不是 5 分钟完成的任务，往往跨天甚至跨周。

所以产品必须支持：

- 会话中断后恢复
- 项目状态持久化
- 步骤级 checkpoint
- 下一次进入时从上次节点继续

这也是为什么它本质上更像“项目智能体”而不是“聊天机器人”。

### 18.4 Structured outputs 必须是默认能力
所有关键产物都必须结构化，而不是靠自然语言后处理。

必须结构化的对象包括：

- 槽位状态
- 证据卡
- 合同步骤
- 检查点
- 高风险警告
- 应急分支
- 评测 trace

### 18.5 Evals 要变成产品开发的一部分
不能只在论文评测时做实验，产品上线后反而靠感觉迭代。

建议建立四层评估：

1. **单能力评测**
   意图识别、槽位抽取、证据召回、计划生成。
2. **流程评测**
   从用户输入到合同输出的完整任务成功率。
3. **轨迹评测**
   每一步工具调用是否合理、是否多问、是否漏问、是否误放行。
4. **线上行为评测**
   用户是否真的执行、是否在安全停步后流失、是否减少返工。

## 20. 什么叫“可用的产品”

对这个项目来说，“可用”不是指技术上能输出一段回答，而是用户真的能拿它去完成一次漆艺任务。

### 20.1 可用产品的最低标准

1. 用户能在 3 分钟内开始一个项目。
2. 系统能看懂“我要做什么”或“我出了什么问题”。
3. 系统不会在关键变量缺失时直接给冒险建议。
4. 系统能产出一份用户看得懂、做得到的步骤清单。
5. 用户能在执行过程中回来继续，而不是重新开一轮聊天。

### 20.2 可用产品的最小闭环

#### 第一步：新建项目
用户输入目标或问题，附带图片可选。

#### 第二步：系统补问
系统只问当前决策真正需要的信息。

#### 第三步：生成执行方案
输出：

- 当前判断
- 下一步怎么做
- 暂时不能做什么
- 需要确认什么
- 做错了怎么补救

#### 第四步：执行与记录
用户边做边勾选，记录环境和结果。

#### 第五步：复盘与复用
项目结束后沉淀成个人模板或教学案例。

### 20.3 第一版最建议上线的产品定义
最适合率先上线的不是“大而全平台”，而是：

**一个面向漆艺学习者的项目式 AI 工作台**

它至少包含 4 个稳定页面：

1. **首页**
   新建“制作项目”或“排障项目”。
2. **项目工作台**
   左侧上下文，中间对话，右侧合同。
3. **执行清单页**
   逐步勾选、暂停、继续。
4. **知识卡页**
   查看依据、边界条件和典型案例。

### 20.4 第一版不要犯的产品错误

- UI 里充满论文术语
- 让用户看到一堆 JSON
- 一上来问十几个问题
- 输出很专业但根本没法执行
- 没有暂停/继续能力
- 没有“现在别做”的明确提示

## 21. 产品化后的表达方式

面向用户时，产品语言应该从“论文机制”改成“行动语言”。

### 21.1 不建议直接暴露的研究语言

- VoI
- slot filling
- irreversibility gate
- executable contract
- retrieval grounding

### 21.2 建议转换成的产品语言

- 关键条件
- 还缺的信息
- 现在先别继续
- 你的执行方案
- 为什么这样建议
- 做错了怎么办

### 21.3 一句话原则
**论文负责解释系统为什么成立，产品负责让用户马上知道下一步该干什么。**
